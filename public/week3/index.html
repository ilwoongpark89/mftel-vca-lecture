더 좋은 문장을 고르는 일은 이미 한계에 닿았다. 모델의 성능을 결정하는 진짜 지렛대는 추론 시점에 무엇을 보여줄 것인가 — 즉 컨텍스트 윈도우 안에 어떤 정보를, 어떤 순서로, 얼마나 넣느냐다. 문서 $N$개 중 상위 $k$개만 골라 넣는 검색 증강 생성(RAG)에서, 관련도 순위와 토큰 예산 $B$의 균형이 답의 정확도를 좌우한다. 나아가 모델이 스스로 도구를 부르고 결과를 되먹임하는 에이전트 루프에 이르면, 프롬프트는 더 이상 한 번의 지시가 아니라 계속 갱신되는 상태(state)가 된다. [Lewis et al., Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks, NeurIPS 2020]

Picking better wording has already hit its ceiling. The real lever on model performance is what you show it at inference time — that is, which information, in what order, and how much you place inside the context window. In retrieval-augmented generation (RAG), where only the top $k$ of $N$ documents are injected, the balance between relevance ranking and a token budget $B$ governs the accuracy of the answer. Push further to an agent loop, where the model calls tools and feeds the results back to itself, and the prompt stops being a single instruction — it becomes a continuously updated state. [Lewis et al., Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks, NeurIPS 2020]

RAG — 모델에게 "열린 책"을 쥐여주기

RAG — Handing the Model an "Open Book"

📚
직관 — 폐쇄형 시험(closed-book)을 보는 학생은 오직 암기에 의존해 답하고, 모르면 그럴듯하게 지어냅니다(환각). 반면 오픈북 시험은 먼저 관련 페이지를 찾고, 그 페이지를 앞에 펼쳐 놓은 채 답을 씁니다. RAG는 LLM을 오픈북 응시자로 바꿉니다 — 질문이 오면 문서를 검색(retrieve)하고, 찾은 근거를 프롬프트에 주입(augment)한 뒤 생성(generate)합니다. (출처: Lewis et al. 2020, "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks," NeurIPS)
📚
Intuition — A student in a closed-book exam relies only on memory and, when unsure, makes things up plausibly (hallucination). An open-book exam instead lets you first find the relevant pages and write the answer with those pages open in front of you. RAG turns an LLM into an open-book test-taker — given a query, it retrieves documents, augments the prompt with the found evidence, then generates. (Source: Lewis et al. 2020, "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks," NeurIPS)

RAG(Retrieval-Augmented Generation)는 매개변수에 암묵적으로 저장된 지식(parametric memory)에, 외부 코퍼스에서 명시적으로 끌어온 지식(non-parametric memory)을 결합하는 구조입니다. 세 단계 — 검색 → 증강 → 생성 — 로 이루어지며, 핵심 하이퍼파라미터는 검색 문서 수 $k$입니다. 이 구조는 지식 컷오프(학습 시점 이후 정보 부재)와 환각을 완화하고, 답변에 출처 근거를 부여합니다.

RAG 파이프라인 세 단계
① 검색 retrieve$ D_k = \text{top-}k\ \operatorname{sim}(q,\,d_i) $
② 증강 augment$ \text{prompt} = [D_k;\ q] $
③ 생성 generate$ y \sim p_\theta(y \mid D_k,\, q) $
핵심 용어
질의 $q$ 사용자 질문 — 검색의 입력
코퍼스 $\{d_i\}$ 검색 대상 외부 문서 집합(knowledge base)
검색 문서 수 $k$ 상위 몇 개 문서를 문맥으로 주입할지
parametric memory 가중치 $\theta$에 저장된 암묵 지식
non-parametric memory 외부 코퍼스의 명시 지식(교체·갱신 가능)
grounding 생성 답변을 검색 근거에 정박(출처 제시)
핵심
RAG는 지식을 가중치에서 분리합니다. 새 사실이 생기면 모델을 재학습할 필요 없이 코퍼스만 갱신하면 됩니다 — 지식 컷오프를 넘어서고, 답변을 실제 문서에 정박(grounding)해 환각을 줄입니다.

RAG (Retrieval-Augmented Generation) combines knowledge stored implicitly in weights (parametric memory) with knowledge pulled explicitly from an external corpus (non-parametric memory). It runs in three stages — retrieve → augment → generate — and its key hyperparameter is the number of retrieved documents $k$. This design mitigates the knowledge cutoff (no information after training) and hallucination, while giving each answer source grounding.

The three-stage RAG pipeline
① retrieve$ D_k = \text{top-}k\ \operatorname{sim}(q,\,d_i) $
② augment$ \text{prompt} = [D_k;\ q] $
③ generate$ y \sim p_\theta(y \mid D_k,\, q) $
Key terms
Query $q$ the user's question — input to retrieval
Corpus $\{d_i\}$ external document set to search (knowledge base)
Retrieved count $k$ how many top documents to inject as context
parametric memory implicit knowledge held in weights $\theta$
non-parametric memory explicit corpus knowledge (swappable, updatable)
grounding anchoring the answer to retrieved evidence (citing sources)
Key
RAG decouples knowledge from weights. When a new fact appears, you need not retrain the model — just update the corpus. This crosses the knowledge cutoff and grounds answers in real documents, reducing hallucination.

검색은 왜 잘 되는가 — 유사도와 주변화

Why Retrieval Works — Similarity and Marginalization

공통 관점: RAG 생성은 어떤 문서를 봤느냐에 대한 확률적 평균입니다. 질의 $q$와 각 문서 $d$를 벡터로 임베딩해 유사도로 상위 $k$개를 고르고, 그 문서들에 조건부 생성을 한 뒤 검색 확률로 가중 평균(주변화)합니다.

Unifying view: RAG generation is a probabilistic average over which documents were seen. We embed the query $q$ and each document $d$ as vectors, pick the top-$k$ by similarity, condition generation on those documents, and marginalize (weighted average) by the retrieval probability.

Step 1 · 임베딩 유사도

검색 점수화
🧲 뜻이 가까운 문서를 끌어당긴다
$ s(q,d) = \dfrac{\mathbf{E}(q)\cdot\mathbf{E}(d)}{\lVert\mathbf{E}(q)\rVert\,\lVert\mathbf{E}(d)\rVert} $

질의·문서를 인코더 $\mathbf{E}$로 벡터화하고 코사인 유사도로 점수를 매깁니다. 점수 상위 $k$개가 $D_k$.

Step 2 · 검색 분포

softmax 정규화
⚖️ 유사도를 확률로 바꾼다
$ p_\eta(d\mid q) = \dfrac{\exp\big(s(q,d)/\tau\big)}{\sum_{d'\in D_k}\exp\big(s(q,d')/\tau\big)} $

상위 $k$ 문서의 유사도를 softmax로 정규화해 각 문서에 검색 확률을 부여. $\tau$는 온도.

Step 3 · 주변화 생성

RAG-Sequence 근사
🎯 여러 근거를 확률로 섞는다
$ p(y\mid q) \approx \sum_{d\in D_k} p_\eta(d\mid q)\, p_\theta(y\mid d,q) $

각 문서 조건 생성 확률을 검색 확률로 가중 합(주변화). $k$를 늘리면 더 많은 근거를 섞습니다.

Step 1 · Embedding similarity

scoring retrieval
🧲 Pull in documents close in meaning
$ s(q,d) = \dfrac{\mathbf{E}(q)\cdot\mathbf{E}(d)}{\lVert\mathbf{E}(q)\rVert\,\lVert\mathbf{E}(d)\rVert} $

Encode query and document with encoder $\mathbf{E}$ and score by cosine similarity. The top-$k$ scores form $D_k$.

Step 2 · Retrieval distribution

softmax normalization
⚖️ Turn similarities into probabilities
$ p_\eta(d\mid q) = \dfrac{\exp\big(s(q,d)/\tau\big)}{\sum_{d'\in D_k}\exp\big(s(q,d')/\tau\big)} $

Normalize the top-$k$ similarities with a softmax to assign each document a retrieval probability. $\tau$ is temperature.

Step 3 · Marginalized generation

RAG-Sequence approx.
🎯 Blend several sources by probability
$ p(y\mid q) \approx \sum_{d\in D_k} p_\eta(d\mid q)\, p_\theta(y\mid d,q) $

Take a weighted sum of per-document generation probabilities, weighted by retrieval probability (marginalization). Larger $k$ blends more evidence.

주의 · $k$는 공짜가 아니다
$k$를 키우면 정답 문서가 포함될 확률(재현율)은 오르지만, 무관 문서도 함께 섞여 문맥을 희석하고 토큰 예산·지연을 늘립니다. 실제로 관련 없는 문맥은 정확도를 떨어뜨릴 수 있습니다(context distraction). 그래서 곡선은 포화·역전 형태입니다. (경향은 대표적 서술이며 절대 수치는 UNVERIFIED)
Caution · $k$ is not free
Increasing $k$ raises the chance the gold document is included (recall), but it also mixes in irrelevant documents that dilute context and cost more tokens and latency. Irrelevant context can in fact lower accuracy (context distraction), so the curve tends to saturate and reverse. (This trend is a representative account; absolute numbers are UNVERIFIED)

Lewis et al. 2020 — RAG의 탄생

Lewis et al. 2020 — the Birth of RAG

2020년 Patrick Lewis와 동료들(Facebook AI Research)은 지식 집약형 과제에서 순수 파라미터 모델의 한계를 지적하며 RAG를 제안했습니다. 핵심 아이디어는 사전학습된 seq2seq 생성기(BART)에 밀집 검색기(DPR)와 위키피디아 벡터 인덱스를 결합해, 검색과 생성을 함께 미세조정(end-to-end)하는 것이었습니다.

지식을 가중치에 가두지 말고, 필요할 때 찾아오게 하라.

논문은 두 변종을 제시합니다 — RAG-Sequence(전체 답변을 하나의 문서에 조건)와 RAG-Token(토큰마다 다른 문서를 참조). 이렇게 분리된 비파라미터 메모리는 재학습 없이 인덱스만 교체해 지식을 갱신할 수 있고, 생성 근거를 문서로 추적(provenance)할 수 있게 합니다. 오늘날 대부분의 실무 RAG는 이 골격을 단순화해 검색 → 문맥 주입 → 생성의 3단으로 씁니다.

꼭 알아야 할 점
  • RAG = parametric(가중치) + non-parametric(코퍼스) 결합.
  • 검색기와 생성기를 end-to-end로 함께 학습 가능(원논문).
  • 실무 최소 골격은 retrieve → augment → generate 3단.

참고: Lewis et al. (2020) NeurIPS / Karpukhin et al. (2020) DPR, EMNLP

In 2020, Patrick Lewis and colleagues (Facebook AI Research) proposed RAG, noting the limits of purely parametric models on knowledge-intensive tasks. The core idea was to combine a pretrained seq2seq generator (BART) with a dense retriever (DPR) and a Wikipedia vector index, and to fine-tune retrieval and generation jointly (end-to-end).

Don't lock knowledge into the weights — let the model fetch it when needed.

The paper offers two variants — RAG-Sequence (the whole answer conditioned on one document) and RAG-Token (each token can attend to a different document). This separated non-parametric memory lets you update knowledge by swapping the index without retraining, and makes generation traceable to its documents (provenance). Most production RAG today simplifies this skeleton to a three-step retrieve → augment → generate.

Must know
  • RAG = parametric (weights) + non-parametric (corpus).
  • Retriever and generator can be trained end-to-end (original paper).
  • Minimal production skeleton is retrieve → augment → generate.

Refs: Lewis et al. (2020) NeurIPS / Karpukhin et al. (2020) DPR, EMNLP

The Dilemma of $k$ — Tuning the Grounding Score

$k$를 늘리면 정답 문서가 포함될 확률(재현율)은 포화 곡선으로 오르지만, 무관 문서가 문맥을 희석해 페널티가 붙습니다. 아래 장난감 모델로 근거성 점수 $G(k)=\big(1-e^{-k/k_0}\big)-\lambda\,k$를 조절해보세요 — 재현항은 포화, 희석항 $\lambda k$는 선형 페널티. (교육용 모형이며 절대 수치는 UNVERIFIED)

Raising $k$ improves recall (chance the gold document is included) as a saturating curve, but irrelevant documents dilute the context and add a penalty. Tune the toy model below: grounding score $G(k)=\big(1-e^{-k/k_0}\big)-\lambda\,k$ — a saturating recall term minus a linear dilution penalty $\lambda k$. (An illustrative model; absolute numbers are UNVERIFIED)

공학적 감각 — RAG에서 $k$는 정확도와 비용의 손잡이입니다. 재현항 $1-e^{-k/k_0}$은 $k\!\gg\!k_0$에서 $1$에 포화 — 더 넣어도 새 정답 정보가 거의 없습니다. 반면 희석·토큰·지연 비용은 $k$에 선형으로 늘죠. 실무 감: 짧은 사실 질의는 $k$가 작게(예 $3\!\sim\!5$), 종합·비교 질의는 크게 잡되 재순위화(re-rank)로 무관 문서를 거릅니다. (모형·경향은 교육용, 절대 수치 UNVERIFIED / 개념 출처: Lewis et al. 2020)
Engineering sense — In RAG, $k$ is the knob between accuracy and cost. The recall term $1-e^{-k/k_0}$ saturates toward $1$ when $k\!\gg\!k_0$ — adding more brings almost no new gold information. Meanwhile dilution, token, and latency costs grow linearly in $k$. Rule of thumb: short factual queries want a small $k$ (e.g. $3\!\sim\!5$); synthesis/comparison queries want more but with re-ranking to filter irrelevant documents. 청킹·재순위·캐싱 — RAG 검색 파이프라인의 세 손잡이

Chunking, Reranking, Caching — Three Knobs of the RAG Pipeline

📚
직관 — 도서관 사서에게 질문할 때, ① 책을 몇 쪽씩 잘라 색인할지(청크 크기), ② 후보 20권을 뽑은 뒤 정말 관련 있는 3권을 다시 골라 위로 올릴지(재순위), ③ 매번 반복하는 서문·규정을 미리 복사해두고 재사용할지(프롬프트 캐싱). 이 세 손잡이가 검색-증강 생성(RAG)의 정확도와 비용·지연을 동시에 결정한다. (출처: Lewis et al. 2020, "Retrieval-Augmented Generation for Knowledge-Intensive NLP", NeurIPS / Anthropic Prompt Caching docs)
📚
Intuition — When you ask a librarian a question: ① how many pages to slice and index a book into (chunk size), ② whether to pull 20 candidates and then re-pick the 3 truly relevant ones to the top (reranking), ③ whether to pre-copy and reuse the repeated preface/rules (prompt caching). These three knobs jointly set the accuracy, cost, and latency of retrieval-augmented generation (RAG). (Source: Lewis et al. 2020, "Retrieval-Augmented Generation for Knowledge-Intensive NLP", NeurIPS / Anthropic Prompt Caching docs)

청킹(chunking)은 문서를 임베딩·검색 단위로 자르는 것입니다. 청크가 너무 작으면 임베딩은 정밀하지만 문맥이 조각나 답에 필요한 앞뒤 정보가 다른 청크로 흩어집니다. 반대로 너무 크면 한 청크에 여러 주제가 섞여 임베딩 신호가 잡음으로 희석됩니다. 재순위(rerank)는 1차 검색(임베딩 유사도)로 뽑은 상위 $k$ 후보를 교차-인코더로 다시 채점해 순서를 바로잡습니다. 프롬프트 캐싱은 반복되는 접두부(prefix)를 재사용해 처리 비용과 지연을 줄입니다.

세 손잡이 요약

핵심
청크·재순위는 정확도 손잡이, 캐싱은 비용·지연 손잡이. 캐싱은 접두부 정확 일치(prefix match)가 조건 — 접두부 한 바이트만 바뀌어도 그 뒤 캐시는 전부 무효화됩니다.

Chunking slices a document into units for embedding and retrieval. Chunks that are too small give precise embeddings but fragment the context, scattering the surrounding information a good answer needs across different chunks. Chunks that are too large mix multiple topics into one, diluting the embedding signal with noise. Reranking takes the top-$k$ candidates from first-stage retrieval (embedding similarity) and re-scores them with a cross-encoder to fix the ordering. Prompt caching reuses the repeated prefix to cut processing cost and latency.

The three knobs
Chunk sizeprecision ↔ context-preservation tradeoff
Rerank $k$recall (wide) → precision (re-sort)
Cache hit rate $h$cost/latency drop ($\text{read} \approx 0.1\times\text{base}$)
Key terms
Chunk an embedding/retrieval unit cut from a document (e.g. 256–1024 tokens)
Overlap tokens shared by adjacent chunks — mitigates boundary context loss
Reranker a cross-encoder that re-scores the $k$ candidates (query+doc jointly)
Recall@k probability the answer is within the top-$k$ candidates
Cache hit rate $h$ fraction of the request prefix reused from cache $[0,1]$
Prefix the front span rendered in order tools → system → messages
Key
Chunking and reranking are accuracy knobs; caching is a cost/latency knob. Caching requires an exact prefix match — a single byte change in the prefix invalidates the entire cache after it.

왜 이렇게 되는가 — 세 메커니즘의 수식

Why It Works — the Math of Three Mechanisms

공통 관점: ① 청크 크기는 정밀도(작을수록↑)와 문맥 보존(클수록↑)의 곱으로 recall이 결정되는 단봉(unimodal) 곡선을 만들고, ② 재순위는 2단계 파이프라인(넓은 recall → 정밀 rerank)으로 계산량을 아끼며, ③ 캐싱은 히트율 $h$에 선형인 유효비용을 만듭니다.

Unifying view: ① chunk size makes a unimodal curve where recall is the product of precision (rises as size↓) and context preservation (rises as size↑), ② reranking saves compute via a two-stage pipeline (wide recall → precise rerank), and ③ caching produces an effective cost linear in the hit rate $h$.

① 청크 크기 곡선

정밀도 × 문맥
🔪 작게 자를수록 임베딩은 날카롭지만 문맥이 흩어진다
$ \text{recall}(s) \approx P_{\text{prec}}(s)\cdot P_{\text{ctx}}(s) $

$P_{\text{prec}}$는 청크가 작을수록↑(신호 집중), $P_{\text{ctx}}$는 클수록↑(문맥 보존). 두 감소·증가 함수의 곱 → 중간에 최댓값을 갖는 단봉 곡선.

② 재순위 2단계

recall → precision
🎯 넓게 20개 뽑고, 무거운 채점은 20개에만
$ C \approx C_{\text{emb}}\!\cdot\! N + C_{\text{ce}}\!\cdot\! k,\ \ k\ll N $

1차: 값싼 임베딩 유사도로 전체 $N$에서 상위 $k$. 2차: 비싼 교차-인코더 $C_{\text{ce}}$를 $k$개에만 적용($k\ll N$)해 precision↑ 하되 계산량 억제.

③ 캐싱 유효비용

히트율 $h$에 선형
📋 반복 접두부는 read 요금(≈0.1×)으로
$ \bar{c}(h) = c_f(1-h) + c_c\,h $

미스분($1-h$)은 정가 $c_f$, 히트분($h$)은 캐시 read $c_c\approx 0.1\,c_f$. $h$가 클수록 유효비용이 선형으로 감소.

① Chunk-size curve

precision × context
🔪 smaller cuts sharpen embeddings but scatter context
$ \text{recall}(s) \approx P_{\text{prec}}(s)\cdot P_{\text{ctx}}(s) $

$P_{\text{prec}}$ rises as size↓ (concentrated signal); $P_{\text{ctx}}$ rises as size↑ (context kept). The product of a decreasing and an increasing function → a unimodal curve with an interior maximum.

② Two-stage rerank

recall → precision
🎯 pull 20 wide; run the heavy scorer on just those 20
$ C \approx C_{\text{emb}}\!\cdot\! N + C_{\text{ce}}\!\cdot\! k,\ \ k\ll N $

Stage 1: cheap embedding similarity picks top-$k$ from all $N$. Stage 2: apply the costly cross-encoder $C_{\text{ce}}$ to only $k$ items ($k\ll N$), raising precision while bounding compute.

③ Cache effective cost

linear in hit rate $h$
📋 repeated prefix billed at read rate (≈0.1×)
$ \bar{c}(h) = c_f(1-h) + c_c\,h $

The miss share $(1-h)$ pays full price $c_f$; the hit share $h$ pays the cache read $c_c\approx 0.1\,c_f$. Larger $h$ → effective cost drops linearly.

주의 · 캐시 write 프리미엄 & 무효화
캐시 write는 정가의 약 1.25×(5분 TTL)입니다. 따라서 접두부를 2번 이상 재사용해야 손익분기($1.25\times + 0.1\times < 2\times$). 접두부에 타임스탬프·UUID가 섞이면 매 요청마다 write만 하고 read=0 → 비용이 오히려 늘어납니다. (출처: Anthropic Prompt Caching — 경제성 · 무효화 규칙. 제공사별 구체 할인율은 UNVERIFIED)
Caution · cache write premium & invalidation
A cache write costs about 1.25× full price (5-minute TTL). So you must reuse the prefix at least twice to break even ($1.25\times + 0.1\times < 2\times$). If a timestamp or UUID leaks into the prefix, every request only writes (read=0) and cost increases. (Source: Anthropic Prompt Caching — economics & invalidation rules. 교차-인코더 재순위의 근거와 캐싱의 프리픽스 불변식

Why Cross-Encoder Reranking, and the Prefix Invariant of Caching

1차 검색의 bi-encoder는 질의와 문서를 따로 벡터로 인코딩해 내적으로 유사도를 잽니다 — 빠르지만 상호작용을 못 봅니다. 재순위기(cross-encoder)는 [질의; 문서]를 함께 넣어 토큰 단위 상호작용을 계산하므로 정밀하지만 느립니다(Nogueira & Cho 2019, "Passage Re-ranking with BERT"). 그래서 전체 $N$은 bi-encoder로 좁히고, 상위 $k$만 cross-encoder로 재채점하는 2단계가 표준입니다.

넓게 뽑아 recall을 확보하고, 좁게 다시 채점해 precision을 얻는다.

프롬프트 캐싱의 유일한 불변식은 접두부 정확 일치입니다. 렌더 순서는 tools → system → messages이며, 캐시 키는 각 breakpoint까지의 정확한 바이트로 결정됩니다. 안정적인 내용(고정 system, 결정적 tool 목록)을 앞에, 변동 내용(질문·per-request ID)을 마지막 breakpoint 뒤에 두어야 합니다. usage.cache_read_input_tokens이 반복 요청에서 0이면 조용한 무효화 범인이 있는 것입니다.

꼭 알아야 할 점
  • 재순위는 recall→precision 2단계 — cross-encoder는 상위 $k$에만.
  • 캐시는 prefix match 불변식 — 앞이 바뀌면 뒤 전부 무효.
  • 손익분기: 접두부 ≥2회 재사용(5분 TTL 기준).
  • read≈$0.1\times$, write≈$1.25\times$(5분) / $2\times$(1시간).

참고: Nogueira & Cho (2019) arXiv:1901.04085 / Lewis et al. (2020) NeurIPS / Anthropic Prompt Caching docs (제공사별 수치는 UNVERIFIED)

First-stage bi-encoders encode query and document into vectors separately and measure similarity by dot product — fast, but they never see the interaction. A cross-encoder reranker feeds [query; document] together, computing token-level interaction — precise but slow (Nogueira & Cho 2019, "Passage Re-ranking with BERT"). Hence the standard two-stage design: narrow all $N$ with the bi-encoder, then re-score only the top-$k$ with the cross-encoder.

Pull wide to secure recall; re-score narrow to earn precision.

Prompt caching has one invariant: an exact prefix match. The render order is tools → system → messages, and the cache key is the exact bytes up to each breakpoint. Put stable content (frozen system, deterministic tool list) first, and volatile content (the question, per-request IDs) after the last breakpoint. If usage.cache_read_input_tokens is zero across repeated requests, a silent invalidator is at work.

Must know
  • Reranking is a recall→precision two-stage — cross-encoder on the top-$k$ only.
  • Caching is a prefix-match invariant — change the front, lose everything after.
  • Break-even: reuse the prefix ≥2× (5-minute TTL).
  • read≈$0.1\times$, write≈$1.25\times$(5m) / $2\times$(1h).

Refs: Nogueira & Cho (2019) arXiv:1901.04085 / Lewis et al. (2020) NeurIPS / Anthropic Prompt Caching docs (provider percentages UNVERIFIED)

Key Points · Common Mistakes · Exam Focus

핵심 정리

  • 청크 크기: 작으면 정밀·문맥손실, 크면 문맥보존·잡음 → recall은 단봉 곡선(중간 최적, 대표 $\approx 512$ 토큰).
  • 재순위: recall→precision 2단계 — 전체 $N$은 bi-encoder, 상위 $k$만 cross-encoder ($k\ll N$).
  • 캐싱: 유효비용 $\bar{c}=c_f(1-h)+c_c h$, 절감 $=h(1-r)$, read $\approx 0.1\times$ · write $\approx 1.25\times$(5분).

⚠ 자주 하는 실수

  • "청크는 작을수록 좋다" 오해 → 문맥손실로 recall↓.
  • 접두부에 타임스탬프·UUID → 매번 write, read=0.
  • 접두부 1회만 쓰고 캐싱 → write 프리미엄만 손해.

시험 포인트

  • 절감률 $=h(1-r)$ 계산 (히트율·read 비율 주고 %).
  • 2단계 재순위 계산량 비 $N/k$.
  • 캐시 무효화(prefix match) 원리 · 손익분기($\ge 2$회 재사용).

Key points

  • Chunk size: small = precise·context loss, large = context kept·noise → recall is a unimodal curve (interior optimum, representative $\approx 512$ tokens).
  • Reranking: a recall→precision two-stage — all $N$ by bi-encoder, only top-$k$ by cross-encoder ($k\ll N$).
  • Caching: effective cost $\bar{c}=c_f(1-h)+c_c h$, saving $=h(1-r)$, read $\approx 0.1\times$ · write $\approx 1.25\times$ (5m).

⚠ Common mistakes

  • Thinking "smaller chunks are always better" → recall drops from context loss.
  • A timestamp/UUID in the prefix → writes every time, read=0.
  • Caching a prefix used only once → you eat the write premium for nothing.

Exam focus

  • Compute saving $=h(1-r)$ (given hit rate & read ratio → %).
  • Two-stage rerank compute ratio $N/k$.
  • Cache invalidation (prefix match) & break-even (reuse $\ge 2\times$).

에이전트 루프 — 계획·실행·검증의 삼박자

The Agent Loop — Plan, Act, Verify in Rhythm

🧭
직관 — 낯선 도시에서 목적지를 찾는다고 해봅시다. 무작정 걷는 게 아니라 지도를 보고 경로를 짜고(계획), 한 블록 걸어가고(실행), 표지판으로 위치를 확인(검증)한 뒤 다시 경로를 고칩니다. LLM 자율 에이전트도 똑같습니다 — 한 번의 생성으로 끝내지 않고, 추론과 행동을 번갈아 반복하며 외부 세계의 피드백으로 자신을 교정합니다. (출처: Yao et al. 2023, "ReAct: Synergizing Reasoning and Acting in Language Models", ICLR 2023)
🧭
Intuition — imagine finding a destination in an unfamiliar city. You don't just walk blindly: you read the map and plan a route (plan), walk one block (act), check your position by the signs (verify), then revise the route. An autonomous LLM agent works the same way — instead of one-shot generation, it interleaves reasoning and action repeatedly, correcting itself with feedback from the outside world. (Source: Yao et al. 2023, "ReAct: Synergizing Reasoning and Acting in Language Models", ICLR 2023)

단발성 프롬프트는 모델의 한 번의 추측에 결과를 겁니다. 반대로 에이전트 루프는 계획(plan) → 실행(act) → 검증(verify)의 세 단계를 하나의 사이클로 묶어 반복합니다. 핵심은 검증 게이트입니다 — 실행 결과가 테스트·컴파일·명세 같은 외부 기준을 통과해야만 다음으로 넘어가고, 실패하면 관측(observation)을 근거로 다시 계획합니다. 코딩 에이전트가 테스트가 통과할 때까지 반복하는 것이 대표적인 형태입니다.

핵심
루프의 성패는 검증 게이트의 존재에 달려 있습니다. 검증이 없으면 모델의 잘못된 관측이 다음 계획의 전제가 되어 오류가 눈덩이처럼 불어납니다(환각 누적). 검증은 루프를 수렴시키는 앵커입니다.

A single-shot prompt bets the outcome on one guess by the model. An agent loop, by contrast, ties three stages — plan → act → verify — into one repeated cycle. The key is the verification gate: the acted result must pass an external criterion (tests, compilation, spec) before proceeding; on failure the agent re-plans using the observation. A coding agent that iterates until tests pass is the canonical instance.

왜 검증이 루프를 수렴시키는가 — 성공확률 모델

Why Verification Makes the Loop Converge — a Success-Probability Model

공통 관점: 한 번의 시도가 성공할 확률을 $q$라 합시다. 검증 게이트가 있으면 실패한 시도는 버려지고 성공한 시도만 채택됩니다 — 그래서 $N$번 반복 시 적어도 한 번 성공할 확률로 품질이 결정됩니다. 검증이 없으면 실패를 걸러내지 못해 마지막 한 번의 결과에 운명을 겁니다.

Unifying view: let one attempt succeed with probability $q$. With a verification gate, failed attempts are discarded and only a passing one is accepted — so quality is set by the probability of at least one success over $N$ tries. Without verification, failures are not filtered and you gamble on the last single result.

① 한 번 시도의 성공률

단일 attempt
🎯 한 발 — 맞을 확률 q, 빗나갈 확률 1−q
$ P_1 = q,\quad P_{\text{fail}} = 1-q $

검증 유무와 무관하게 한 번의 계획-실행이 정답을 낼 확률은 $q$입니다.

② 검증 있는 N회 반복

gate ON
🔁 통과할 때까지 — 모두 실패해야 최종 실패
$ P_{\text{pass}}(N) = 1-(1-q)^{N} $

게이트가 실패를 걸러내므로, $N$번 중 단 한 번만 통과해도 채택됩니다. $N$이 커지면 $1$로 수렴.

③ 검증 없는 N단계

gate OFF
☠️ 오류 전파 — 각 단계가 앞 단계에 의존
$ P_{\text{ok}}(N) = q^{N} $

검증이 없으면 모든 단계가 옳아야 최종이 맞습니다. 반복이 늘수록 오히려 $0$으로 붕괴(환각 누적).

① Success of one attempt

single attempt
🎯 One shot — hit q, miss 1−q
$ P_1 = q,\quad P_{\text{fail}} = 1-q $

Regardless of verification, one plan-act cycle produces a correct result with probability $q$.

② N tries with verification

gate ON
🔁 Until it passes — all must fail to fail
$ P_{\text{pass}}(N) = 1-(1-q)^{N} $

The gate filters failures, so a single passing try among $N$ is accepted. As $N$ grows it converges to $1$.

③ N steps without verification

gate OFF
☠️ Error propagation — each step depends on the last
$ P_{\text{ok}}(N) = q^{N} $

Without verification every step must be right. More iterations collapse toward $0$ (hallucination compounding).

주의 · 반복이 항상 좋은 것은 아니다
$1-(1-q)^{N}$ 모델은 검증 게이트가 완벽하다고 가정합니다. 실제로는 테스트가 불완전하면(거짓 통과) 잘못된 결과가 게이트를 뚫고, 검증 없는 루프에서는 반복이 오류를 증폭시킵니다($q^{N}\!\to\!0$). 즉 반복 자체가 아니라 "믿을 수 있는 검증"이 품질의 원천입니다.
Caution · more iterations aren't always better
The $1-(1-q)^{N}$ model assumes a perfect gate. In reality, an incomplete test suite lets wrong results slip through (false pass), and an ungated loop amplifies errors ($q^{N}\!\to\!0$). So the source of quality is "trustworthy verification," not iteration itself.

ReAct에서 Reflexion으로 — 관측과 성찰의 힘

From ReAct to Reflexion — the Power of Observation and Reflection

ReAct(Yao et al. 2023)는 언어모델이 추론(reasoning) 흔적과 행동(action)을 번갈아 생성하도록 하여, 순수 추론만 할 때 발생하는 사실 환각을 외부 도구의 관측으로 줄였습니다. 저자들은 이 interleaving이 계획을 유연하게 조정하고 예외를 처리하게 만든다고 보고했습니다.

추론만으로는 환각하고, 행동만으로는 계획하지 못한다 — 둘을 엮어야 한다.

Reflexion(Shinn et al. 2023, NeurIPS)은 여기에 자기 성찰을 더했습니다. 에이전트가 실패하면 그 원인을 언어적 피드백으로 요약해 메모리에 저장하고, 다음 시도의 프롬프트에 넣어 같은 실수를 반복하지 않도록 합니다. 가중치를 갱신하는 강화학습 대신, 텍스트 피드백으로 학습하는 것이 특징입니다. 코딩 벤치마크에서 반복적 자기수정이 단발 생성보다 통과율을 높였다고 보고되었습니다. (정확한 수치는 원논문 표 참조 — 본 강의에서는 UNVERIFIED로 둡니다)

꼭 알아야 할 점
  • ReAct = reasoning ⨯ acting의 interleaving으로 관측을 계획에 되먹임.
  • Reflexion = 실패를 언어 피드백으로 요약해 다음 시도에 주입 (가중치 갱신 없음).
  • 공통 앵커 = 외부 신호(관측·테스트)가 루프를 교정한다.

참고: Yao et al. (2023) ICLR / Shinn et al. (2023) "Reflexion", NeurIPS / SWE-bench(Jimenez et al. 2024)

ReAct (Yao et al. 2023) has the language model alternate reasoning traces and actions, reducing the factual hallucination of reasoning-only prompting by grounding on observations from external tools. The authors report that this interleaving lets the model flexibly adjust plans and handle exceptions.

Reasoning alone hallucinates; acting alone cannot plan — the two must be interwoven.

Reflexion (Shinn et al. 2023, NeurIPS) adds self-reflection. When the agent fails, it summarizes the cause as verbal feedback, stores it in memory, and injects it into the next attempt's prompt so it avoids repeating the mistake. Notably it learns from text feedback rather than updating weights via reinforcement learning. On coding benchmarks, iterative self-correction was reported to raise pass rates over single-shot generation. (For exact figures see the original tables — treated here as UNVERIFIED.)

Must know
  • ReAct = interleaving reasoning ⨯ acting, feeding observations back into the plan.
  • Reflexion = summarize failure as verbal feedback injected next try (no weight update).
  • Common anchor = external signal (observation/test) corrects the loop.

Q1.What is the essential role of the verification gate in an agent loop?

Explanation Answer ② — the gate judges pass/fail by an external criterion (tests, type-check), discarding failures and accepting only successes. Wrong ①·③: ① length is irrelevant; ③ even a perfect gate with finite $N$ isn't $100\%$, and an imperfect gate admits false passes. class n=41 · ① 5 · ② 31 · ③ 5

Q2.검증 없이 여러 단계를 이어 붙이면 최종 성공확률은 왜 $q^{N}$으로 붕괴하는가?

해설 정답 ① — 게이트가 없으면 실패를 걸러내지 못해 각 단계가 앞 단계에 의존한다. 독립 가정 하에 $N$단계가 모두 옳을 확률은 $q\times q\times\cdots=q^{N}$이고, $q<1$이면 $N$이 커질수록 $0$으로 붕괴한다(환각 누적). 오답 ②·③: 반복 자체가 지능이나 정확도를 높이는 것이 아니며, 오히려 검증이 없으면 오류가 증폭된다. 학급 n=41 · ① 30 · ② 6 · ③ 5

핵심 정리 · 자주 하는 실수 · 시험 포인트

Key Points · Common Mistakes · Exam Focus

핵심 정리

  • 에이전트 루프 = 계획(plan) → 실행(act) → 검증(verify) → 반복의 사이클.
  • 검증 게이트 ON: $P=1-(1-q)^{N}\to 1$ (실패를 걸러 수렴).
  • 검증 OFF: $P=q^{N}\to 0$ (오류 전파 = 환각 누적).
  • 코딩 에이전트 = 테스트 통과까지 반복; ReAct·Reflexion 계열.

⚠ 자주 하는 실수

  • 반복 횟수 자체를 품질로 착각 → 진짜 앵커는 믿을 수 있는 검증.
  • 불완전한 테스트의 거짓 통과를 무시 → 게이트 품질이 상한을 정한다.
  • ReAct(추론+행동)와 Reflexion(언어 성찰)을 혼동 → 후자는 가중치 갱신 없음.

시험 포인트

  • $q$, $N$ 주고 $1-(1-q)^{N}$ vs $q^{N}$ 계산·대비 문제 빈출.
  • 검증 게이트가 왜 루프를 수렴시키는가(실패 필터링) 서술.
  • ReAct(Yao 2023)·Reflexion(Shinn 2023)의 핵심 아이디어와 차이.

Key points

  • Agent loop = the cycle plan → act → verify → repeat.
  • Gate ON: $P=1-(1-q)^{N}\to 1$ (filter failures, converge).
  • Gate OFF: $P=q^{N}\to 0$ (error propagation = hallucination compounding).
  • Coding agent = iterate until tests pass; ReAct·Reflexion family.

⚠ Common mistakes

  • Mistaking iteration count for quality → the real anchor is trustworthy verification.
  • Ignoring false passes from incomplete tests → gate quality sets the ceiling.
  • Confusing ReAct (reason+act) with Reflexion (verbal reflection) → the latter has no weight update.

Exam focus

  • Given $q$, $N$: compute and contrast $1-(1-q)^{N}$ vs $q^{N}$ (frequent).
  • Explain why a gate makes the loop converge (failure filtering).
  • Core ideas and differences of ReAct (Yao 2023) & Reflexion (Shinn 2023).

명세 먼저, 코드 나중 — 그리고 역할을 나눈 에이전트들

Spec First, Code Later — and Agents That Divide the Work

🏗️
직관 — 집을 지을 때 도면·검수 기준 없이 벽부터 올리면, 나중에 문이 안 맞아 다시 허물게 됩니다. "바이브 코딩(vibe coding)"으로 LLM에게 곧장 코드를 시키면 초기 속도는 빠르지만 무엇이 맞는지의 기준(수용기준)이 없어 재작업이 쌓입니다. 명세 주도 개발은 도면(명세)과 검수표(수용기준)를 먼저 확정하고, 큰 공사는 설계자·시공자·감리자로 역할을 나누듯 멀티 에이전트로 나눕니다. (출처: IEEE 830-1998 SRS 권고 / Boehm, Software Engineering Economics, 1981 — 결함 조기 발견 원리)
🏗️
Intuition — Building a house by stacking walls with no blueprint or inspection checklist means the doors won't fit later, forcing you to tear it back down. "Vibe coding" — asking an LLM to write code immediately — is fast at first, but with no standard of correctness (acceptance criteria) the rework piles up. Spec-driven development fixes the blueprint (specification) and inspection checklist (acceptance criteria) first, and — as big projects split into architect, builder, inspector — divides work across multiple agents. (Source: IEEE 830-1998 SRS guidance / Boehm, Software Engineering Economics, 1981 — early-defect-detection principle)

명세 주도(spec-driven)란 코드를 쓰기 전에 ① 요구 명세(무엇을) → ② 수용기준(어떻게 검증) → ③ 구현(코드) 순서를 강제하는 방식입니다. 프롬프트 관점에서는 "이걸 만들어줘"가 아니라 "이 명세와 수용기준을 만족하는 코드를 작성하라"로 지시가 바뀝니다. 명세가 검증 가능한 테스트로 번역되면, LLM 출력은 "그럴듯함"이 아니라 통과/실패로 판정됩니다.

왜 명세가 재작업을 줄이는가 — 부채의 수식

Why Specs Cut Rework — the Debt Equation

공통 관점: 재작업은 "산출물이 기준과 어긋난 채 다음 단계로 넘어간 양"에 비례합니다. 명세가 없으면 기준 자체가 모호해 어긋남이 뒤늦게 발견되고, 발견이 늦을수록 되돌릴 코드가 많아집니다. 명세 완성도 $s$를 높이면 어긋남의 기대값이 줄어 재작업이 감소합니다.

Unifying view: rework is proportional to "how much output passed to the next stage while misaligned with the standard." Without a spec the standard is vague, misalignment is caught late, and the later it is caught the more code must be undone. Raising spec completeness $s$ lowers the expected misalignment and thus rework.

명세 완성도 → 재작업

$ R(s) = k\left(1-\tfrac{s}{100}\right) $

$s$: 명세 완성도(%), $k$: 명세 전무($s{=}0$) 시 기대 재작업 횟수. 선형 골격 — $s{\to}100$이면 $R{\to}0$.

$ C \approx c_0\, m^{\,d} $

$d$: 결함이 흘러간 단계 수, $m{>}1$: 단계당 배율. Boehm(1981)의 "늦게 잡을수록 비싸다" 정성 관계. 배율 값은 UNVERIFIED.

$\Delta Q$: 역할 분리로 얻는 품질 이득, $\alpha(n{-}1)$: 에이전트 $n$개의 조율·통합 비용. $G{\le}0$이면 단일이 낫다.

$\Delta Q$: quality gain from role separation, $\alpha(n{-}1)$: coordination/integration cost of $n$ agents. If $G{\le}0$, a single agent is better.

주의 · 선형 모델의 한계
$R(s)=k(1-s/100)$은 직관용 골격일 뿐 실측 상수가 아닙니다($k$, 선형성 모두 UNVERIFIED). 실제로는 명세 완성도가 매우 높아도 사양 자체의 오류·모호성이 남아 잔여 재작업 $R_{min}{>}0$이 존재합니다. 또한 "멀티 에이전트가 항상 낫다"는 일반적 우위는 입증되지 않았습니다(UNVERIFIED) — 태스크 의존성·조율 비용에 따라 단일이 우세할 수 있습니다.
Caution · limits of the linear model
$R(s)=k(1-s/100)$ is only an intuition skeleton, not a measured constant (both $k$ and linearity are UNVERIFIED). In practice, even with very high completeness, errors/ambiguity in the spec itself leave a residual rework $R_{min}{>}0$. Likewise, a general claim that "multi-agent is always better" is UNVERIFIED — depending on task dependency and coordination cost, a single agent can win.

명세는 오래된 지혜, 에이전트는 새 무대

Specs Are Old Wisdom; Agents Are the New Stage

수용기준과 검증 가능한 요구는 소프트웨어공학에서 오래 다뤄졌습니다. IEEE 830-1998은 좋은 요구 명세의 성질로 정확·모호하지 않음·검증 가능·완전·일관을 제시했고, 행위 주도 개발(BDD)은 이를 given/when/then 시나리오로 실행 가능한 명세로 만들었습니다(North, 2006). 바이브 코딩 시대에도 원리는 같습니다 — 검증할 수 없는 요구는 프롬프트로도 검증할 수 없습니다.

"검증할 수 없으면 요구가 아니다." — 명세의 제1원칙

멀티 에이전트는 새 무대입니다. 토론·역할극이 추론을 개선한다는 보고가 있으나(예: Du et al., 2023, multiagent debate; Wu et al., AutoGen, 2023), 단일 강한 에이전트 + 좋은 명세가 더 낫거나 대등한 경우도 흔합니다. 태스크가 분해 가능하고 병렬적이면 역할 분리가 유리하고, 단계가 강하게 의존하면 조율 비용이 이득을 잠식합니다(태스크 의존에 따른 일반 우위는 UNVERIFIED).

꼭 알아야 할 점
  • 명세는 검증 가능한 수용기준으로 번역돼야 값이 생긴다.
  • 테스트가 곧 "완료의 정의" — LLM 출력의 합격 판정선.
  • 멀티 에이전트의 우위는 조건부 — 조율 비용을 항상 계산하라.

참고: IEEE Std 830-1998 / D. North, "Introducing BDD"(2006) / Du et al.(2023) arXiv:2305.14325 / Wu et al., AutoGen(2023) arXiv:2308.08155

Acceptance criteria and verifiable requirements are long-standing in software engineering. IEEE 830-1998 lists the qualities of a good requirement spec as correct, unambiguous, verifiable, complete, consistent; behavior-driven development (BDD) turned these into executable specs via given/when/then scenarios (North, 2006). The principle survives into the vibe-coding era — a requirement you cannot verify, you cannot verify with a prompt either.

"If you can't verify it, it isn't a requirement." — first principle of specs

Multi-agent is the new stage. Debate and role-play are reported to improve reasoning (e.g., Du et al., 2023, multiagent debate; Wu et al., AutoGen, 2023), yet a single strong agent + a good spec is often better or on par. When a task is decomposable and parallel, role separation helps; when stages are tightly dependent, coordination cost erodes the gain (a general advantage by task dependency is UNVERIFIED).

Must know
  • A spec earns its value only when translated into verifiable acceptance criteria.
  • Tests are the "definition of done" — the pass line for LLM output.
  • Multi-agent advantage is conditional — always price the coordination cost.

Refs: IEEE Std 830-1998 / D. North, "Introducing BDD" (2006) / Du et al. (2023) arXiv:2305.14325 / Wu et al., AutoGen (2023) arXiv:2308.08155
